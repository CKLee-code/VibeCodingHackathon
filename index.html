Evaluation Design — 2-Phase System
Phase 1: Intra-country peer selection → Phase 2: Global leadership final judging
PHASE 1 — Peer Evaluation
▸ Who: All participants within the same country
▸ What: Score any submitted project in their country lane
▸ Constraint: Cannot score own team
▸ Result: Normalized average score → Country Rank #1
▸ Output: 1 finalist per country = 7 finalists total
→
PHASE 2 — Leadership Final
▸ Who: 14 Leadership Evaluators + Approved Volunteer Judges
▸ What: Score all 7 country finalist projects
▸ Constraint: Cannot evaluate own country's finalist
▸ Quorum: Minimum 7 evaluators required per project
▸ Result: Average of all participating evaluator scores
▸ Output: Global Final Ranking (1st–7th)

Fairness Mechanisms — Unknown Participation Rate
The system must produce fair results regardless of how many teams submit or how many participants vote
⚠️

Core Challenge: Vietnam has only 1 team (1 pair = 2 people). Philippines has 33 teams. If peer voting is a raw average, a country with 1 evaluator vs. 30 evaluators creates structurally different reliability. The following mechanisms address this.

Phase 1 — Peer Score Calculation Formula

Final_Score(project) = Σ(score_i) / N → Normalized to 100pt scale
Country_Winner = argmax( Normalized_Avg_Score ) per country
Where N = number of peer evaluations received for that project. Minimum N threshold applies (see below). If a country has only 1 submitted project → that project automatically advances as country representative.

Phase 2 — Final Evaluation Formula

Global_Score(project) = Σ(evaluator_score_j) / N where N = actual participating evaluators
Final_Rank = rank() OVER (ORDER BY Global_Score DESC)
Evaluators: 14 Leadership + Approved Volunteer Judges. Minimum quorum = 7 evaluators per project. Cannot evaluate own country's finalist. Tie-breaking: higher Business Impact score wins; if still tied → higher Technical Execution score.
4 Fairness Mechanisms
Applied to Phase 1 peer evaluation to ensure robustness

📐 Mechanism 1 — Minimum Evaluator Threshold

A project's peer score is only counted if it receives at least 2 evaluations from different participants. This prevents a single evaluator from determining a country winner unilaterally.

If fewer than 2 evaluations are received for any project, admin must manually trigger a reminder to country participants before Phase 1 closes.

MIN_EVALS = 2 per project
Exception: Vietnam (1 team) → auto-finalist

📊 Mechanism 2 — Outlier Score Trimming

If a project receives 5 or more evaluations, the highest and lowest single scores are dropped before averaging. This guards against strategic inflating (voting up friends) or deflating (voting down competitors).

Applied per project, not per criterion. Trimming only activates at N≥5 to avoid distorting small samples.

N ≥ 5: trim max + min score
Final = Σ(remaining) / (N-2)

🚫 Mechanism 3 — Self-Vote Prevention (DB Enforced)

A database-level CHECK constraint prevents any team member from evaluating their own team's project. This is enforced at both the application layer (UI hides own project from evaluation list) and database layer (constraint violation throws error).

Evaluator's country must match the project's country for peer evaluation. Cross-country peer voting is blocked.

CHECK: evaluator_user_id NOT IN
SELECT user_id FROM team_members
WHERE team_id = project.team_id

🔍 Mechanism 4 — Blind Evaluation Display

During Phase 1 peer evaluation, the evaluator sees only the project title, description, demo URL, and video. Team name and team member names are hidden until after evaluation is submitted.

This reduces social/hierarchy bias — junior staff are less likely to inflate scores for senior colleagues' projects when names are hidden.

Reveal: team_name shown AFTER
peer_eval.status = 'submitted'

Evaluation Timeline
Aligned with the 3-day hackathon schedule (SGT)

DAY 2 — Code Freeze

  • Submission window closes at 23:59 SGT
  • All projects locked to 'submitted' status
  • Peer evaluation window opens immediately after freeze
  • Participants can view and evaluate all submitted projects in their country

DAY 3 — Peer Results

  • Peer evaluation window closes at 15:00 SGT
  • Scores_Summary view refreshed
  • Country #1 finalists automatically determined
  • Finalist status updated in PROJECTS table
  • 14 evaluators notified of their assessment queue

DAY 3 — Final Judging

  • Leadership evaluation window opens at 15:00 SGT
  • All 14 judges score all 7 finalists
  • Global ranking computed in real-time
  • Results revealed at Awards Ceremony
  • Final leaderboard published on website
